## 🔴 阻塞项（8 项）

### 🔴-1 连续亏损未按 exit_date 排序，多 symbol 时结果错误
- 文件：backend/app/api/analysis.py:168, 106-121
- 已核实：`_compute_consecutive_losses(valid_positions)` 直接遍历未排序的列表。同文件 line 236 的 max_drawdown 用了 `sorted(valid_positions, key=lambda p: p.exit_date)`，唯独 consecutive_losses 没排序。多 symbol 时 positions 按 symbol 分组返回，非全局按 exit_date 排序，连续亏损数算错。
- 建议：`_compute_consecutive_losses(sorted(valid_positions, key=lambda p: p.exit_date))`

### 🔴-2 上传先读后校验致内存耗尽 DoS
- 文件：backend/app/api/upload.py:49-50
- 已核实：`content = file.file.read()` 在 `if len(content) > MAX_UPLOAD_BYTES` 之前全量读入内存。Content-Length 头是客户端可控的，可省略或伪造。攻击者发 1GB 文件先分配 1GB 内存再被拒绝，并发上传致 OOM。
- 建议：`content = file.file.read(MAX_UPLOAD_BYTES + 1)` 限制读取字节数

### 🔴-3 AI 报告生成端点无限流 — 财务 DoS
- 文件：backend/app/api/report.py:71-76
- 已核实：`/api/report/generate` 调用外部 LLM API 产生真实费用，无 `@limiter.limit`。已认证用户可无限次调用，短时间产生数千美元账单。前两轮只给 auth/admin 加了限流，遗漏业务端点。
- 建议：加 `@limiter.limit("3/minute")`

### 🔴-4 PYRAMID/AVERAGE_DOWN 仍用全局首尾买入价，跨周期比较错误
- 文件：backend/app/engine/pattern.py:176-177
- 已核实：`first_buy_price = same_symbol_trades[0].price` 和 `last_buy_price = same_symbol_trades[-1].price` 取该 symbol 全部交易的首尾买入价，非当前持仓周期。buy→sell→buy→sell 场景下跨周期比较，误标/漏标。第二轮修复了 AVERAGE_DOWN 的亏损判断（🔴-8），但 first_buy_price 的根本问题未修。
- 建议：按 pos 的 trade_ids 或 entry_date~exit_date 范围过滤出当前仓位的买入交易

### 🔴-5 AVERAGE_DOWN 外层条件引用错误的 first_buy_price + 内层缺 0.95 阈值
- 文件：backend/app/engine/pattern.py:200-219
- 已核实：外层 `if last_buy_price < first_buy_price * 0.95` 用全局 first_buy_price（🔴-4 的根因），内层 `last_buy_price < prev_pos.avg_entry_price` 只检查"低于均价"缺 FINANCE_DOMAIN.md 定义的 ×0.95 阈值。
- 建议：外层改用当前周期首买价，内层改用 `last_buy_price < prev_pos.avg_entry_price * 0.95`

### 🔴-6 validator _find_pnl_for_pattern 正则误匹配次数而非盈亏
- 文件：backend/app/ai/validator.py:97-102
- 已核实：正则 `{escaped}.*?([+-]?\d+\.?\d*)` 用 re.DOTALL 匹配模式名后第一个数字，但 prompt 格式是"BREAKOUT: 5次, 胜率60%, 总盈亏+1234.56"，第一个数字是次数 5 不是盈亏 1234.56。校验总不匹配 → 触发无意义重试 → validation_passed=False。
- 建议：锚定到盈亏上下文 `r"{escaped}.*?(?:总?盈亏|pnl)[：:\s]*([+-]?\d+\.?\d*)"`

### 🔴-7 validator fallback 正则匹配任意数字
- 文件：backend/app/ai/validator.py:44-46, 58-62
- 已核实：交易次数 fallback `r"(\d+)"` 匹配任意数字（年份 2024、日期等），胜率 fallback `r"(\d+(?:\.\d+)?)%"` 匹配任意百分比。LLM 未用"交易次数"原词时，fallback 匹配到年份 2024 与 total_trades=50 比较，判定不匹配。
- 建议：移除 fallback 或缩窄到 "N笔" / "N次交易" 等变体，找不到就跳过

### 🔴-8 report.py 缺 psychology 标签 + tag_position 未传 trades
- 文件：backend/app/api/report.py:94-99
- 已核实：generate_report 未调用 detect_psychological_patterns，AI 报告缺心理维度。tag_position 未传 trades，TURN/PYRAMID/AVERAGE_DOWN 依赖 trades 参数，不传时回退到低精度推断或丢失。对比 analysis.py 的 _build_category_map 完整调用了。
- 建议：复用 _build_category_map，或至少补齐 trades 参数和 psychology 调用

## 🟡 建议项（12 项）

### 🟡-1 TURN 标签未按 position 日期范围过滤 trades
- 文件：backend/app/engine/pattern.py:251-273
- 原因：same_symbol_trades 未按 pos.entry/exit 过滤，该 symbol 只要做过一次T，所有仓位都打 TURN

### 🟡-2 FOMO 检测 entry_idx==5 时 dates[i-1] 回绕到列表末尾
- 文件：backend/app/engine/pattern.py:443-449
- 原因：i=0 时 dates[-1] 回绕，用错误的 close 做比较。改为 `entry_idx >= 6` 或循环改为 `range(entry_idx-4, entry_idx)`

### 🟡-3 TURN 被 SCALP/SWING/POSITION 恒定覆盖，实际从不生效
- 文件：backend/app/engine/pattern.py:74-84
- 原因：TURN 置信度 0.7 < SCALP/SWING/POSITION 的 1.0，同在 behavior 维度被 resolve_per_category 覆盖。做T行为永远不出现在分析中
- 建议：TURN 提升到独立维度或特殊处理

### 🟡-4 HOLD_LOSER/CUT_WINNER 用 pnl < 0 而非 pnl <= 0
- 文件：backend/app/engine/pattern.py:690-691
- 已核实：全项目用 pnl <= 0（含平本），唯独此处用 pnl < 0，pnl==0 的仓位被排除在中位数计算外

### 🟡-5 list_reports 对手机号注册用户崩溃
- 文件：backend/app/api/report.py:207
- 已核实：`current_user.email.split("@")[0]`，email 为 None 时 AttributeError → 500。User.email 是 nullable 的

### 🟡-6 resolve_hierarchy 仅在有 market_data 时调用，BREAKOUT+CHASE 去重丢失
- 文件：backend/app/api/report.py:96-98, analysis.py:356-360
- 原因：BREAKOUT+CHASE 去重不依赖 market_data，但被放在 `if market_data:` 块内。无行情数据时同一仓位被双重计入归因

### 🟡-7 provider.py 每次请求创建新 AsyncOpenAI 客户端未关闭，连接池泄漏
- 文件：backend/app/ai/provider.py:24-30
- 原因：get_llm() 每次创建新实例，从不 close()。httpx.AsyncClient 连接池泄漏
- 建议：缓存为模块级单例

### 🟡-8 ensure_market_data 在 async 端点中同步阻塞事件循环
- 文件：backend/app/api/report.py:90
- 原因：generate_report 是 async def 但调同步的 ensure_market_data，N 个缺失 symbol 串行阻塞事件循环
- 建议：`await asyncio.to_thread(ensure_market_data, ...)`

### 🟡-9 schemas 缺少输入校验（日期范围、字段长度）
- 文件：backend/app/schemas/analysis.py:9-13
- 原因：AnalysisRunRequest 不校验 date_end >= date_start，filename 无长度限制

### 🟡-10 上传和分析端点缺失限流
- 文件：backend/app/api/upload.py:30, analysis.py:74
- 原因：/api/upload 无限流可耗尽 DB 存储，/api/analysis/run 无限流可创建大量记录

### 🟡-11 CORS origin 硬编码，生产不可配置
- 文件：backend/app/main.py:30-36
- 原因：allow_origins=["http://localhost:5173"] 硬编码，生产需改代码或误设 "*"

### 🟡-12 raw_content 明文存储敏感金融数据
- 文件：backend/app/models/raw_file.py:20
- 原因：用户完整交易记录明文 LargeBinary 存储，DB 泄露即全量暴露
